Accessibility commitment

Accessibility

FOCUS is being built for blind and sighted creators. The goal is WCAG 2.2 AA conformance, with special attention to keyboard use, screen reader support, readable copy, and predictable production workflows.

What is already built into the interface

  • Semantic headings, landmarks, tables, and native controls.
  • A skip link that moves keyboard focus to the main content.
  • Visible focus indicators for keyboard users.
  • Form labels, help text, error summaries, and invalid field states.
  • Descriptive link and button text that names the affected item.
  • Live regions for sign-in status, copy actions, messages, and new notifications.

Testing still needed before production

The project still needs broader manual testing with NVDA, JAWS, Narrator, VoiceOver, keyboard-only navigation, browser zoom, and high contrast or forced colors. Automated accessibility checks should also be added to continuous integration before launch.

Known scope

Public trust pages, sign-in flows, account safety pages, group pages, member pages, project pages, notifications, and destructive confirmation pages should all be part of the production accessibility review. Machine endpoints, downloads, and development-only routes should be tracked separately from user-facing pages.
